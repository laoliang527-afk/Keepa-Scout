# 技术报告 — Amazon Arbitrage Scout

## 技术选型说明

### 数据库：SQLite

我们选了 SQLite，因为它完全够用——目前只有几十个商品，数据结构简单，没有多人同时写入的需求。用了异步驱动（aiosqlite），FastAPI 服务不会卡住。

如果以后商品数量涨到1万以上，或者需要多台服务器同时访问，那就换成 PostgreSQL，代码几乎不用改。

### AI 模型：DeepSeek Chat

DeepSeek 是国产大模型，能力接近 GPT-4，但 API 费用只有 GPT-4 的几十分之一。

我们的做法是**两步走**：
1. 第一步：让 AI 根据问题生成 SQL 查询语句
2. 第二步：把数据库里真实查出来的数据，交给 AI 格式化成自然语言回答

这样做的好处是：AI 只能回答数据库里**真实存在**的数据，不会瞎编数字。

### Web 框架：FastAPI + 异步

全程用异步操作（`httpx` 发请求、`aiosqlite` 查数据），一个 API 请求卡住不会影响其他请求的响应速度。

## AI 回答质量的迭代过程

### 第一版（单步 prompt）

最早的方案是：让 AI **一步到位**——同时生成 SQL、执行 SQL、再生成回答，全部塞进一个 prompt 里。

结果出了两个问题：

1. **AI 瞎编答案**：AI 生成了一条听起来合理但实际错误的 SQL，然后围绕那个错误结果一本正经地回答，普通人完全看不出来
2. **SQL 注入风险**：SQL 语句直接混在 AI 输出里，很难安全地校验

### 第二版（两步 pipeline）

拆成两段：

1. **第一步**：专门让 AI 生成 SQL，prompt 里只有数据库结构和一个 JSON 输出格式要求，temperature=0.1（非常确定性的输出），保证每次生成的 SQL 格式稳定
2. **第二步**：把真实查到的数据（行数列名都摆在 AI 面前），再让 AI 格式化回答。AI 没法凭空捏造，因为数据就在 prompt 里

**结果**：第二版彻底消灭了瞎编数字的问题，所有回答都基于真实数据库结果。

## AI 工具使用说明

- **使用的 AI 工具**：Cursor（IDE 内嵌 AI 助手）
- **大模型**：DeepSeek `deepseek-chat` 用于自然语言转 SQL；`cursor-agent`（Claude-4 系列）用于代码实现
- **代码来源**：所有源代码由 AI 助手编写，用户审核和指导方向。关键决策（两步 pipeline、SQLite 选型、复合索引）由用户和 AI 共同讨论确定

## 如果有更多时间，会做这些改进

1. **批量 eligibility 自动拆分**：如果一次输入150+个ASIN，自动拆成小批次并发请求
2. **AI 回答缓存**：同一个问题重复问时，直接返回缓存结果，省 API 调用费用
3. **完整的 `/chat` 6种上下文模式**：实现"阈值替换"（"改成30%"）、用户偏好自动注入到 SQL、"第二个"这类指代词的精准解析
4. **增量 ETL 更新**：加一个"上次更新时间"字段，每次只刷新超过N小时没更新的商品，而不是每次全量重新拉
5. **更准确的 UPC-E 展开**：用真正的校验位算法做 UPC-E → UPC-A 展开，而不是目前简化的对应表
